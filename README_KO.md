## 팁 0: 상태 표시줄 커스터마이징

Claude Code 하단의 상태 표시줄을 유용한 정보가 표시되도록 커스터마이징할 수 있습니다. 저는 모델명, 현재 디렉토리, git 브랜치(있는 경우), 커밋되지 않은 파일 수, origin과의 동기화 상태, 토큰 사용량 시각적 진행 표시줄이 보이도록 설정했습니다. 또한 마지막 메시지를 보여주는 두 번째 줄도 표시해서 대화 주제가 무엇이었는지 확인할 수 있습니다:

```
Opus 4.5 | 📁claude-code-tips | 🔀main (scripts/context-bar.sh uncommitted, synced 12m ago) | ██░░░░░░░░ 18% of 200k tokens
💬 This is good. I don't think we need to change the documentation as long as we don't say that the default color is orange el...
```

컨텍스트 사용량을 주시하고 작업 중이던 내용을 기억하는 데 특히 유용합니다. 스크립트는 10가지 컬러 테마(orange, blue, teal, green, lavender, rose, gold, slate, cyan, gray)도 지원합니다.

## 팁 1: 필수 슬래시 명령어 익히기

내장된 슬래시 명령어가 많이 있습니다(`/`를 입력하면 전체 목록을 볼 수 있습니다). 알아두면 좋은 몇 가지:

### /usage

사용량 한도 확인:

```
 Current session
 █████████▌                                         19% used
 Resets 12:59am (America/Vancouver)
```

사용량을 면밀히 모니터링하려면 탭에 열어두고 Tab → Shift+Tab 또는 ← → 로 새로고침하세요.

### /chrome

Claude의 기본 브라우저 통합 켜기/끄기:

```
> /chrome
Chrome integration enabled
```

### /mcp

MCP(Model Context Protocol) 서버 관리:

```
 Manage MCP servers
 1 server
 ❯ 1. playwright  ✔ connected · Enter to view details
```

### /stats

GitHub 스타일 활동 그래프로 사용 통계 확인:

```
  Favorite model: Opus 4.5        Total tokens: 17.6m
  Sessions: 4.1k                  Longest session: 20h 40m 45s
  Active days: 79/80              Longest streak: 75 days
```

### /clear

대화를 지우고 새로 시작합니다.

## 팁 2: 음성으로 Claude Code와 대화하기

타이핑보다 음성으로 훨씬 빠르게 소통할 수 있다는 것을 발견했습니다. 로컬 머신에서 음성 인식 시스템을 사용하면 매우 유용합니다.

Mac에서 사용해 본 옵션들:
- [superwhisper](https://superwhisper.com/)
- [MacWhisper](https://goodsnooze.gumroad.com/l/macwhisper)
- [Super Voice Assistant](https://github.com/ykdojo/super-voice-assistant) (오픈소스, Parakeet v2/v3 지원)

호스팅 서비스를 사용하면 더 높은 정확도를 얻을 수 있지만, 로컬 모델도 이 용도에는 충분합니다. 음성 인식에 오류나 오타가 있어도 Claude는 의도를 파악할 만큼 똑똑합니다.

흔한 반론은 "다른 사람이 있는 방에서는 어떡하냐?"입니다. 저는 이어폰을 사용해 속삭입니다 - 개인적으로 Apple EarPods(AirPods 아님)를 좋아합니다. 저렴하고 품질도 충분하며, 조용히 속삭이기만 하면 됩니다. 이 방법은 비행기에서도 작동할 정도로 효과적입니다.

## 팁 3: 큰 문제를 작게 쪼개기

마스터해야 할 가장 중요한 개념 중 하나입니다. 전통적인 소프트웨어 엔지니어링과 정확히 같습니다 - 최고의 소프트웨어 엔지니어는 이미 이 방법을 알고 있으며, Claude Code에도 동일하게 적용됩니다.

Claude Code가 어려운 문제를 한 번에 해결하지 못한다면, 여러 개의 작은 이슈로 분해하도록 요청하세요. 개별 부분을 해결할 수 있는지 확인하세요. 여전히 어렵다면 더 작은 하위 문제로 나누세요. 모든 것이 해결 가능해질 때까지 계속하세요.

즉, A에서 B로 직접 가는 대신:

A → A1 → A2 → A3 → B로 갈 수 있습니다.

중요한 점: 문제 해결 능력과 소프트웨어 엔지니어링 스킬은 에이전트 코딩과 Claude Code 세계에서도 여전히 매우 중요합니다. Claude Code가 많은 문제를 스스로 해결할 수 있지만, 여러분의 일반적인 문제 해결 및 소프트웨어 엔지니어링 스킬을 적용하면 훨씬 강력해집니다.

## 팁 4: Git과 GitHub CLI 프로처럼 사용하기

Git과 GitHub CLI 작업은 그냥 Claude에게 맡기세요. 커밋(커밋 메시지를 직접 작성하지 않아도 됨), 브랜칭, 풀, 푸시 등이 포함됩니다.

저는 개인적으로 pull은 자동 승인하지만 push는 하지 않습니다. push가 더 위험하기 때문입니다 - pull은 문제가 생겨도 origin을 오염시키지 않습니다.

GitHub CLI(`gh`)로 할 수 있는 일이 많습니다. Claude Code 사용 후 더 자주 하게 된 것 중 하나는 draft PR 생성입니다. Claude Code가 PR 생성 과정을 처리하되 위험은 낮게 - 리뷰 준비 표시 전에 모든 것을 검토할 수 있습니다.

`gh`는 꽤 강력합니다. 임의의 GraphQL 쿼리도 보낼 수 있어서, GitHub PR 설명이 편집된 정확한 시간도 찾을 수 있습니다.

### 커밋/PR 출처 표시 비활성화

기본적으로 Claude Code는 커밋에 `Co-Authored-By` 트레일러를, PR에 출처 표시 푸터를 추가합니다. `~/.claude/settings.json`에 다음을 추가하면 둘 다 비활성화할 수 있습니다:

```json
{
  "attribution": {
    "commit": "",
    "pr": ""
  }
}
```

## 팁 5: AI 컨텍스트는 우유와 같다 - 신선하고 농축된 것이 최고!

Claude Code와 새 대화를 시작할 때 성능이 가장 좋습니다. 대화 이전 부분의 컨텍스트를 처리해야 하는 복잡성이 없기 때문입니다. 대화가 길어질수록 컨텍스트가 길어지고 성능이 저하되는 경향이 있습니다.

따라서 새 주제마다, 또는 성능이 저하되기 시작하면 새 대화를 시작하는 것이 가장 좋습니다.

## 팁 6: 터미널에서 출력 내보내기

Claude Code의 출력을 복사-붙여넣기하고 싶을 때가 있지만, 터미널에서 직접 복사하면 항상 깔끔하지는 않습니다. 콘텐츠를 더 쉽게 내보내는 방법들:

- **`/copy` 명령어**: 가장 간단한 옵션 - `/copy`를 입력하면 Claude의 마지막 응답이 마크다운으로 클립보드에 복사됩니다
- **클립보드 직접 사용**: Mac이나 Linux에서 Claude에게 `pbcopy`로 출력을 클립보드에 직접 보내도록 요청
- **파일에 쓰기**: Claude에게 콘텐츠를 파일에 넣고 VS Code(또는 선호하는 에디터)에서 열도록 요청. 특정 줄 번호를 지정할 수도 있습니다
- **URL 열기**: 직접 확인하고 싶은 URL이 있다면, Claude에게 브라우저에서 열어달라고 요청
- **GitHub Desktop**: Claude에게 현재 저장소를 GitHub Desktop에서 열도록 요청

## 팁 7: 빠른 접근을 위한 터미널 별칭 설정

Claude Code 때문에 터미널을 더 많이 사용하게 되면서, 짧은 별칭을 설정하면 빠르게 실행할 수 있어 유용합니다:

- `c` - Claude Code (가장 많이 사용)
- `ch` - Chrome 통합이 포함된 Claude Code
- `gb` - GitHub Desktop
- `co` - VS Code
- `q` - 대부분의 프로젝트가 있는 프로젝트 디렉토리로 이동

셸 설정 파일(`~/.zshrc` 또는 `~/.bashrc`)에 다음과 같은 줄을 추가하세요:

```bash
alias c='claude'
alias ch='claude --chrome'
alias gb='github'
alias co='code'
alias q='cd ~/Desktop/projects'
```

별칭에 플래그를 조합할 수 있습니다: `c -c`는 마지막 대화를 이어가고, `c -r`은 최근 대화 목록을 보여줍니다.

## 팁 8: 적극적으로 컨텍스트 압축하기

Claude Code에는 대화를 요약해서 컨텍스트 공간을 확보하는 `/compact` 명령어가 있습니다. 전체 사용 가능한 컨텍스트가 채워지면 자동 압축도 발생합니다. Opus 4.5의 총 사용 가능한 컨텍스트 윈도우는 현재 200k이며, 그 중 45k는 자동 압축용으로 예약되어 있습니다. 하지만 직접 적극적으로 수행하고 수동으로 조정하는 것이 더 좋다고 느꼈습니다. `/config`으로 auto-compact를 끄고 메인 대화에 더 많은 컨텍스트를 사용하며 압축 시기와 방법을 더 잘 제어합니다.

제가 하는 방법은 새로 시작하기 전에 Claude에게 인수인계 문서를 작성하도록 요청하는 것입니다:

> 나머지 계획을 system-prompt-extraction 폴더에 HANDOFF.md로 작성해. 무엇을 시도했고, 무엇이 효과가 있었고, 무엇이 효과가 없었는지 설명해서, 새로운 컨텍스트를 가진 다음 에이전트가 그 파일만 로드하고도 이 작업을 시작하고 마무리할 수 있도록 해.

그런 다음 새 대화를 시작합니다. 새 에이전트에게는 파일 경로만 주면 됩니다:

```
> experiments/system-prompt-extraction/HANDOFF.md
```

**대안: 플랜 모드 사용**

또 다른 옵션은 플랜 모드입니다. `/plan` 또는 Shift+Tab으로 진입합니다. Claude에게 관련 컨텍스트를 모두 수집하고 다음 에이전트를 위한 포괄적인 계획을 만들도록 요청하세요.

## 팁 9: 자율 작업을 위한 작성-테스트 사이클 완성하기

Claude Code가 `git bisect` 같은 것을 자율적으로 실행하려면, 결과를 검증할 방법을 제공해야 합니다. 핵심은 작성-테스트 사이클을 완성하는 것입니다: 코드 작성, 실행, 출력 확인, 반복.

인터랙티브 터미널이 필요한 작업에는 tmux를 사용할 수 있습니다:

1. tmux 세션 시작
2. 명령어 전송
3. 출력 캡처
4. 기대한 결과인지 검증

### 창의적인 테스트 전략

작성-테스트 사이클을 완성하는 방법에서 창의성이 필요할 때가 있습니다. 예를 들어 웹 앱을 만들고 있다면 Playwright MCP, Chrome DevTools MCP, 또는 Claude의 기본 브라우저 통합(`/chrome`)을 사용할 수 있습니다.

Playwright는 접근성 트리(구조화된 페이지 요소 데이터)에 초점을 맞추는 반면, Claude의 기본 브라우저 통합은 스크린샷을 찍고 특정 좌표의 요소를 클릭하는 데 초점을 맞춥니다. 시각적으로 집중적이지 않은 대부분의 작업에는 기본적으로 Playwright를 사용하는 것을 추천합니다.

## 팁 10: Cmd+A와 Ctrl+A는 좋은 친구

몇 년 전부터 이야기해왔습니다: AI 세계에서 Cmd+A와 Ctrl+A는 친구입니다. Claude Code에도 마찬가지입니다.

Claude Code가 URL에 직접 접근할 수 없을 때가 있습니다. 비공개 페이지이거나, Reddit 게시물처럼 Claude Code가 가져오기 어려운 경우입니다. 이럴 때는 보이는 콘텐츠를 모두 선택(Mac: Cmd+A, 기타: Ctrl+A)하고, 복사한 뒤, Claude Code에 직접 붙여넣으면 됩니다.

## 팁 11: 차단된 사이트 대안으로 Gemini CLI 사용하기

Claude Code의 WebFetch 도구는 Reddit 같은 특정 사이트에 접근할 수 없습니다. 하지만 Gemini CLI를 대안으로 사용하는 스킬을 만들어 해결할 수 있습니다. Gemini는 웹 접근이 가능하며 Claude가 직접 접근할 수 없는 사이트의 콘텐츠를 가져올 수 있습니다.

팁 9의 동일한 tmux 패턴을 사용합니다 - 세션 시작, 명령어 전송, 출력 캡처.

## 팁 12: 자신만의 워크플로우에 투자하기

개인적으로 Swift로 처음부터 음성 인식 앱을 만들고, Claude Code로 커스텀 상태 표시줄을 bash로 만들고, Claude Code의 압축된 JavaScript 파일에서 시스템 프롬프트를 단순화하는 시스템을 만들었습니다.

하지만 그렇게까지 할 필요는 없습니다. CLAUDE.md를 관리하고 가능한 간결하면서도 목표 달성에 도움이 되도록 유지하는 것 - 그런 것들이 도움됩니다. 이런 팁, 도구, 중요한 기능들을 배우는 것도요.

여러분이 만들고 싶은 것을 만들기 위해 사용하는 도구에 대한 투자입니다. 최소한 약간의 시간은 투자할 가치가 있다고 생각합니다.

## 팁 13: 대화 기록 검색하기

Claude Code에게 과거 대화에 대해 물어볼 수 있으며, 찾아서 검색하는 것을 도와줍니다. 대화 기록은 `~/.claude/projects/`에 로컬로 저장되며, 프로젝트 경로를 기반으로 한 폴더명(슬래시가 대시로 변환)을 가집니다.

기본 bash 명령어로 검색할 수 있습니다:

```bash
# "Reddit"을 언급한 모든 대화 찾기
grep -l -i "reddit" ~/.claude/projects/-Users-yk-Desktop-projects-*/*.jsonl

# 대화에서 사용자 메시지만 추출 (jq 필요)
cat ~/.claude/projects/.../conversation-id.jsonl | jq -r 'select(.type=="user") | .message.content'
```

또는 Claude Code에게 직접 물어보세요: "오늘 X에 대해 무슨 이야기했지?"

## 팁 14: 터미널 탭으로 멀티태스킹하기

여러 Claude Code 인스턴스를 실행할 때, 체계적으로 정리하는 것이 Git worktree 같은 특정 기술 설정보다 중요합니다. 한 번에 최대 3~4개의 작업에 집중하는 것을 권장합니다.

제 개인적인 방법은 "캐스케이드"라고 부르는 것입니다 - 새 작업을 시작할 때마다 오른쪽에 새 탭을 엽니다. 그런 다음 왼쪽에서 오른쪽으로 훑으며, 가장 오래된 작업에서 최신 작업으로 이동합니다.

## 팁 15: 시스템 프롬프트 슬림하게 줄이기

Claude Code의 시스템 프롬프트와 도구 정의는 작업을 시작하기도 전에 약 19k 토큰(200k 컨텍스트의 ~10%)을 차지합니다. 이를 약 9k 토큰으로 줄이는 패치 시스템을 만들어 약 10,000 토큰(~50% 오버헤드)을 절약했습니다.

| 구성 요소 | 이전 | 이후 | 절약량 |
|-----------|------|------|--------|
| 시스템 프롬프트 | 3.0k | 1.8k | 1,200 토큰 |
| 시스템 도구 | 15.6k | 7.4k | 8,200 토큰 |
| **합계** | **~19k** | **~9k** | **~10k 토큰 (~50%)** |

더 로우한 느낌입니다 - 더 강력하지만 약간 덜 규제된 느낌인데, 시스템 지시가 짧아져서 당연합니다. 더 적은 컨텍스트로 시작하면 채워지기 전에 더 많은 여유가 있어서 대화를 좀 더 길게 이어갈 수 있습니다.

### MCP 도구 지연 로딩

MCP 서버를 사용한다면, 도구 정의가 기본적으로 모든 대화에 로드됩니다 - 사용하지 않더라도요. 필요할 때만 로드되도록 지연 로딩을 활성화하세요:

```json
{
  "env": {
    "ENABLE_TOOL_SEARCH": "true"
  }
}
```

## 팁 16: 병렬 브랜치 작업을 위한 Git worktree

같은 프로젝트에서 동시에 여러 작업을 하면서 충돌을 피하고 싶다면, Git worktree가 좋은 방법입니다. Claude Code에게 git worktree를 생성하고 거기서 작업을 시작하라고 하면 됩니다 - 구체적인 문법을 걱정할 필요 없습니다.

기본 아이디어는 다른 디렉토리에서 다른 브랜치로 작업할 수 있다는 것입니다. 본질적으로 브랜치 + 디렉토리입니다.

## 팁 17: 장시간 작업을 위한 수동 지수 백오프

Docker 빌드나 GitHub CI 같은 장시간 작업을 기다릴 때, Claude Code에게 수동 지수 백오프를 요청할 수 있습니다. 지수 백오프는 소프트웨어 엔지니어링에서 흔한 기법이지만 여기에도 적용할 수 있습니다. Claude Code에게 점점 늘어나는 대기 간격으로 상태를 확인하도록 요청하세요 - 1분, 2분, 4분 순으로.

## 팁 18: 글쓰기 도우미로서의 Claude Code

Claude Code는 훌륭한 글쓰기 어시스턴트이자 파트너입니다. 제가 글쓰기에 사용하는 방법은 먼저 작성하려는 내용에 대한 모든 컨텍스트를 제공한 다음, 음성으로 상세한 지시를 하는 것입니다. 그렇게 첫 번째 초안을 얻습니다.

그런 다음 거의 줄 단위로 검토합니다. "이 줄은 이런 이유로 좋아", "이 줄은 저기로 옮겨야 할 것 같아", "이 줄은 이렇게 바꿔야 해" 같은 주고받기 과정입니다.

## 팁 19: 마크다운은 최고다

보통 새 문서를 작성할 때 Google Docs나 Notion을 사용할 수 있지만, 이제는 가장 효율적인 방법이 마크다운이라고 생각합니다.

마크다운은 AI 이전에도 꽤 좋았지만, Claude Code와 함께 사용하면 그 가치가 더 높아집니다. 블로그 글이나 LinkedIn 게시물을 쓰고 싶을 때, Claude Code와 대화하고 마크다운으로 저장한 다음 거기서부터 진행할 수 있습니다.

빠른 팁: 마크다운을 쉽게 받지 않는 플랫폼에 복사-붙여넣기하려면, 먼저 새 Notion 파일에 붙여넣은 다음 Notion에서 다른 플랫폼으로 복사하세요.

## 팁 20: Notion을 활용한 링크 보존 붙여넣기

반대 방향도 작동합니다. Slack 같은 곳에서 링크가 있는 텍스트를 복사해서 Claude Code에 직접 붙여넣으면 링크가 보이지 않습니다. 하지만 먼저 Notion 문서에 넣은 다음 거기서 복사하면 마크다운 형태로 얻을 수 있습니다.

## 팁 21: 장시간 위험한 작업을 위한 컨테이너

일반 세션은 권한을 제어하고 출력을 더 신중하게 검토하는 체계적인 작업에 적합합니다. 컨테이너화된 환경은 `--dangerously-skip-permissions` 세션에 적합하며, 각각의 작은 일에 권한을 부여하지 않아도 됩니다.

리서치나 실험 등 시간이 오래 걸리고 위험할 수 있는 작업에 유용합니다. 컨테이너에서 실행하면 문제가 생겨도 격리됩니다.

[SafeClaw](https://github.com/ykdojo/safeclaw)를 만들어서 컨테이너화된 Claude Code 세션을 쉽게 실행할 수 있도록 했습니다.

### 고급: 컨테이너에서 워커 Claude Code 오케스트레이션

로컬 Claude Code가 컨테이너 내부에서 실행되는 다른 Claude Code 인스턴스를 제어하도록 할 수 있습니다. 트릭은 tmux를 제어 레이어로 사용하는 것입니다:

1. 로컬 Claude Code가 tmux 세션을 시작
2. tmux 세션에서 컨테이너에 연결
3. 컨테이너 내부에서 `--dangerously-skip-permissions`로 Claude Code 실행
4. 외부 Claude Code가 `tmux send-keys`로 프롬프트를 보내고 `capture-pane`으로 출력을 읽음

### 고급: 멀티 모델 오케스트레이션

Claude Code 이외에도 다른 AI CLI를 컨테이너에서 실행할 수 있습니다 - Codex, Gemini CLI 등. Claude Code가 다양한 모델을 구동하고 컨테이너와 호스트 간 데이터를 전송하는 등 모든 것을 조율하는 중앙 인터페이스가 됩니다.

## 팁 22: Claude Code를 잘 쓰는 최고의 방법은 직접 쓰는 것

최근 세계적인 암벽 등반 선수가 인터뷰에서 "어떻게 하면 암벽 등반을 잘 할 수 있나요?"라는 질문에 "암벽 등반을 하면 됩니다"라고 답했습니다.

Claude Code도 마찬가지입니다. 영상 보기, 책 읽기, 팁 배우기 등 보충적으로 할 수 있는 것들이 있지만, Claude Code를 사용하는 것이 Claude Code를 배우는 가장 좋은 방법입니다.

10,000시간 법칙 대신 10억 토큰 법칙이라고 생각합니다. AI를 잘 다루고 작동 방식에 대한 좋은 직관을 얻고 싶다면, 많은 토큰을 소비하는 것이 최선입니다.

## 팁 23: 대화 복제/포크와 반복제

대화의 특정 지점에서 원래 스레드를 잃지 않고 다른 접근 방식을 시도하고 싶을 때가 있습니다.

**내장 대안(최근 버전):** Claude Code에는 기본 포킹 기능이 있습니다:
- `/fork` - 대화 내에서 현재 세션을 포크
- `--fork-session` - `--resume` 또는 `--continue`와 함께 사용 (예: `claude -c --fork-session`)

### 컨텍스트 줄이기 위한 반복제

대화가 너무 길어지면, 후반부만 유지하는 반복제 스크립트가 있습니다. 최근 작업을 보존하면서 토큰 사용량을 줄입니다.

### 훅으로 반복제 자동 제안

선택적으로, 컨텍스트가 너무 길어지면 자동으로 `/half-clone`을 트리거하는 훅을 사용할 수 있습니다. 자동 압축보다 좋은 점은 반복제가 결정론적이고 빠르다는 것입니다 - 메시지를 요약하는 대신 실제 메시지를 그대로 유지합니다.

## 팁 24: realpath로 절대 경로 얻기

다른 폴더의 파일에 대해 Claude Code에게 알려줘야 할 때, `realpath`로 전체 절대 경로를 얻으세요:

```bash
realpath some/relative/path
```

## 팁 25: CLAUDE.md vs Skills vs 슬래시 명령어 vs 플러그인 이해하기

비슷한 기능들이라 처음에는 꽤 혼란스러웠습니다. 제가 이해한 내용을 공유합니다.

**CLAUDE.md**는 가장 간단합니다. 매 대화 시작 시 기본 프롬프트로 로드되는 파일들입니다. 프로젝트(`./CLAUDE.md`)나 전역(`~/.claude/CLAUDE.md`)에서 프로젝트 설명 등을 작성할 수 있습니다.

**Skills**는 더 잘 구조화된 CLAUDE.md 파일 같습니다. 관련이 있을 때 Claude가 자동으로 호출하거나, 사용자가 슬래시(예: `/my-skill`)로 수동 호출할 수 있습니다. 필요할 때만 로드되므로 토큰 효율적입니다.

**슬래시 명령어**는 지시를 별도로 패키징하는 방식으로 스킬과 비슷합니다. 사용자가 수동으로 호출하거나 Claude 자체가 호출할 수 있습니다. 더 정확하게, 자신만의 속도로 적절한 시점에 호출해야 할 때 사용합니다.

**플러그인**은 스킬, 슬래시 명령어, 에이전트, 훅, MCP 서버를 함께 패키징하는 방법입니다. 플러그인 형식이 설치를 더 쉽게 만들어줍니다.

## 팁 26: 인터랙티브 PR 리뷰

Claude Code는 PR 리뷰에 탁월합니다. `gh` 명령어로 PR 정보를 가져오도록 요청한 다음, 원하는 방식으로 리뷰할 수 있습니다.

일반적인 리뷰를 하거나, 파일별로, 단계별로 진행할 수 있습니다. 속도를 제어합니다. 얼마나 상세하게, 어떤 수준의 복잡도로 살펴볼지 결정합니다.

핵심 차이점은 Claude Code가 원샷 머신이 아닌 인터랙티브 PR 리뷰어라는 것입니다.

## 팁 27: 리서치 도구로서의 Claude Code

Claude Code는 모든 종류의 리서치에 놀라울 정도로 뛰어납니다. 본질적으로 Google이나 딥 리서치 대체제이지만 여러 면에서 더 고급입니다. GitHub Actions 실패 원인 조사, Reddit에서의 감성/시장 분석, 코드베이스 탐색, 공개 정보 검색 등을 할 수 있습니다.

핵심은 적절한 정보와 접근 방법에 대한 지시를 제공하는 것입니다.

실제로 [Claude Code를 리서치에 활용해 $10,000를 절약](content/how-i-saved-10k-with-claude-code.md)하기도 했습니다.

## 팁 28: 출력 검증의 다양한 방법 마스터하기

코드라면 테스트를 작성하도록 하고 테스트가 전반적으로 괜찮은지 확인하는 방법이 있습니다. Claude Code UI에서 생성되는 코드를 직접 확인할 수도 있습니다. GitHub Desktop 같은 시각적 Git 클라이언트를 사용하는 것도 좋습니다. Draft PR을 생성하게 하고 실제 PR로 전환하기 전에 내용을 확인하는 것도 좋은 방법입니다.

자체 검증도 가능합니다. 리서치 결과를 주면 "이게 확실해? 다시 확인해볼래?"라고 할 수 있습니다. 제가 좋아하는 프롬프트 중 하나는 "모든 것을 다시 확인하고, 생성한 모든 주장을 검증하고, 마지막에 검증할 수 있었던 것을 표로 만들어줘"입니다.

## 팁 29: DevOps 엔지니어로서의 Claude Code

GitHub Actions CI 실패가 있을 때마다, Claude Code에게 주고 "이 이슈를 파고들어서 근본 원인을 찾아봐"라고 합니다. 때로는 피상적인 답을 주지만, 계속 물어보면 - 특정 커밋 때문인지, 특정 PR 때문인지, flaky 이슈인지? - 수동으로는 파고들기 어려운 까다로운 이슈를 정말 잘 해결합니다.

이 워크플로우를 `/gha` 슬래시 명령어로 패키징했습니다 - GitHub Actions URL과 함께 `/gha <url>`을 실행하면 자동으로 실패를 조사하고, flakiness를 확인하고, 원인 커밋을 식별하고, 수정을 제안합니다.

## 팁 30: CLAUDE.md는 간결하게 유지하고 주기적으로 검토하기

CLAUDE.md를 간결하고 가능한 한 축약해서 유지하는 것이 중요합니다. CLAUDE.md 없이 시작해도 됩니다. Claude Code에게 같은 말을 반복적으로 하게 된다면, 그때 CLAUDE.md에 추가하세요.

CLAUDE.md 파일을 주기적으로 검토하는 것도 중요합니다. 시간이 지나면 구식이 될 수 있기 때문입니다.

## 팁 31: 범용 인터페이스로서의 Claude Code

Claude Code는 컴퓨터, 디지털 세계, 모든 종류의 디지털 문제에 대한 범용 인터페이스입니다. 많은 경우 Claude Code에게 맡기면 알아서 해결합니다.

예를 들어 빠른 영상 편집이 필요하면 ffmpeg으로 처리합니다. 오디오/비디오 파일을 트랜스크립션하려면 Python의 Whisper를 제안합니다. CSV 파일의 데이터를 분석하려면 Python이나 JavaScript로 시각화를 제안합니다. 인터넷 접근 - Reddit, GitHub, MCP - 가능성은 무한합니다.

로컬 컴퓨터에서의 작업에도 훌륭합니다. 저장 공간이 부족하면 조언을 구하세요. 로컬 폴더와 파일을 살펴보고, 많은 공간을 차지하는 것을 찾고, 정리 방법을 조언해줍니다. 무엇을 하든 이제 Claude Code가 첫 번째 목적지입니다.

## 팁 32: 올바른 추상화 수준 선택이 핵심

때로는 바이브 코딩 수준에 머물러도 괜찮습니다. 일회성 프로젝트나 코드베이스의 중요하지 않은 부분이라면 모든 코드 줄을 걱정할 필요가 없습니다. 하지만 다른 때는 더 깊이 파고들어야 합니다 - 파일 구조와 함수, 개별 코드 줄, 심지어 의존성까지 확인합니다.

이진법이 아닙니다. 바이브 코딩이 완전히 괜찮을 때도 있지만, 더 깊이 파고들어야 할 때도 있습니다. Claude Code를 가이드 삼아 거대한 빙산을 탐험하는 것과 같습니다.

## 팁 33: 승인된 명령어 감사하기

최근 누군가의 Claude Code가 `rm -rf tests/ patches/ plan/ ~/`를 실행해서 홈 디렉토리를 삭제한 [사례](https://www.reddit.com/r/ClaudeAI/comments/1pgxckk/)를 봤습니다. 이런 실수는 누구에게나 일어날 수 있으므로, 승인된 명령어를 수시로 감사하는 것이 중요합니다. 이를 쉽게 하기 위해 **cc-safe**를 만들었습니다 - `.claude/settings.json` 파일에서 위험한 승인 명령어를 스캔하는 CLI입니다.

```bash
npm install -g cc-safe
cc-safe ~/projects
```

GitHub: [cc-safe](https://github.com/ykdojo/cc-safe)

## 팁 34: 테스트를 많이 작성하기 (그리고 TDD 활용하기)

Claude Code로 더 많은 코드를 작성하면 실수가 생기기 쉬워집니다. PR 리뷰와 시각적 Git 클라이언트가 이슈를 잡는 데 도움되지만, 코드베이스가 커질수록 테스트 작성이 중요합니다.

TDD(테스트 주도 개발)가 Claude Code와 정말 잘 맞습니다:

1. 테스트를 먼저 작성
2. 실패하는지 확인
3. 테스트를 커밋
4. 테스트를 통과하는 코드 작성

먼저 실패하는 테스트를 작성하고 구현 전에 커밋함으로써, 코드가 무엇을 해야 하는지에 대한 명확한 계약을 만듭니다.

## 팁 35: 미지의 영역에서 더 용감하게 - 반복적 문제 해결

Claude Code를 집중적으로 사용하면서, 미지의 영역에서 점점 더 용감해지고 있다는 것을 알았습니다.

예를 들어 [Daft](https://github.com/Eventual-Inc/Daft)에서 일하기 시작했을 때, 프론트엔드 코드에 문제를 발견했습니다. React 전문가는 아니지만, 코드베이스와 문제에 대해 질문을 계속하면서 결국 해결할 수 있었습니다. Claude Code로 반복적으로 문제를 해결하는 방법을 알고 있었기 때문입니다.

핵심은 Claude Code가 초기 솔루션을 제시하고, 좋지 않으면 속도를 늦추고, 다양한 방향을 탐색하고, 때로는 "이 줄이 정확히 무슨 뜻이지?"라고 물으며 추상화 수준과 속도를 제어하는 것입니다.

## 팁 36: 백그라운드에서 bash 명령어와 서브에이전트 실행하기

Claude Code에서 장시간 실행되는 bash 명령어가 있을 때, Ctrl+B를 눌러 백그라운드로 이동시킬 수 있습니다. Claude Code는 백그라운드 프로세스를 관리하는 방법을 알고 있으며, 나중에 BashOutput 도구로 확인할 수 있습니다.

### 서브에이전트 전략적 사용

서브에이전트는 분해해야 할 큰 작업이 있을 때 유용합니다. 거대한 코드베이스를 분석해야 한다면, 서브에이전트가 코드베이스의 다른 부분을 병렬로 분석하도록 할 수 있습니다.

서브에이전트 커스터마이징:
- **수량** - 원하는 만큼 생성 요청
- **백그라운드 vs 포그라운드** - 백그라운드 실행 요청 또는 Ctrl+B
- **모델** - 작업 복잡도에 따라 Opus, Sonnet, Haiku 선택 (서브에이전트 기본값은 Sonnet)

## 팁 37: 개인 맞춤형 소프트웨어의 시대가 왔다

개인 맞춤형 소프트웨어의 시대에 접어들고 있습니다. AI가 나온 이후 - 특히 Claude Code - 때로는 개인용으로, 때로는 작은 프로젝트를 위해 훨씬 더 많은 소프트웨어를 만들 수 있게 되었습니다.

또 다른 예: 인기 있는 Slack MCP인 [korotovsky/slack-mcp-server](https://github.com/korotovsky/slack-mcp-server)는 Docker 컨테이너로 실행되도록 설계되었습니다. 제 Docker 컨테이너 안에서 이를 원활하게 사용하는 데 어려움이 있었습니다(Docker-in-Docker 복잡성). 그래서 Slack의 Node SDK를 직접 사용하는 CLI를 만들어달라고 Claude Code에게 요청했고, 정말 잘 작동했습니다.

무엇이든 원하는 것을 Claude Code에게 요청할 수 있습니다. 충분히 작다면 한두 시간 만에 만들 수 있습니다.

## 팁 38: 입력창 탐색 및 편집하기

Claude Code의 입력창은 일반적인 터미널/readline 단축키를 에뮬레이트하도록 설계되었습니다:

**탐색:**
- `Ctrl+A` - 줄 시작으로 이동
- `Ctrl+E` - 줄 끝으로 이동
- `Option+좌/우` (Mac) 또는 `Alt+좌/우` - 단어 단위 이동

**편집:**
- `Ctrl+W` - 이전 단어 삭제
- `Ctrl+U` - 커서부터 줄 시작까지 삭제
- `Ctrl+K` - 커서부터 줄 끝까지 삭제
- `Ctrl+C` / `Ctrl+L` - 현재 입력 지우기
- `Ctrl+G` - 외부 에디터에서 프롬프트 열기 (긴 텍스트 붙여넣기에 유용)

**줄바꿈 입력(여러 줄 입력):**
가장 빠른 방법: `\` 입력 후 Enter로 줄바꿈. 키보드 단축키는 Claude Code에서 `/terminal-setup` 실행.

**이미지 붙여넣기:**
- `Ctrl+V` (Mac/Linux) 또는 `Alt+V` (Windows) - 클립보드에서 이미지 붙여넣기

참고: Mac에서는 `Cmd+V`가 아니라 `Ctrl+V`입니다.

## 팁 39: 계획에 시간을 들이되 프로토타입도 빠르게

Claude Code가 무엇을 어떻게 만들어야 하는지 알 수 있도록 충분한 시간을 계획에 투자해야 합니다. 어떤 기술을 사용할지, 프로젝트를 어떻게 구조화할지, 각 기능이 어디에 있어야 하는지 등 고수준 결정을 일찍 내려야 합니다.

때로는 프로토타이핑이 도움됩니다. 간단한 프로토타입을 빠르게 만들면 "이 기술이 이 용도에 적합하다" 또는 "다른 기술이 더 낫다"고 판단할 수 있습니다.

Shift+Tab으로 플랜 모드를 사용하거나, 코드를 작성하기 전에 계획을 세우도록 Claude Code에게 요청할 수 있습니다.

## 팁 40: 과도하게 복잡한 코드 단순화하기

Claude Code가 때때로 과도하게 복잡하게 만들고 너무 많은 코드를 작성하는 경우가 있습니다. 요청하지 않은 변경을 합니다. 더 많은 코드를 작성하려는 편향이 있는 것 같습니다. 코드가 정확하게 작동하더라도 유지보수하고 검토하기 어려울 수 있습니다.

때로는 코드를 확인하고 단순화를 요청해야 합니다. "왜 이 특정 변경을 했어?" 또는 "왜 이 줄을 추가했어?"라고 질문할 수 있습니다.

AI로만 코드를 작성하면 절대 이해할 수 없다고 말하는 사람들이 있지만, 충분히 질문하지 않을 때만 그렇습니다. 모든 것을 이해하도록 하면, 대규모 프로젝트에서 특히 AI에게 물어볼 수 있기 때문에 코드를 더 빨리 이해할 수 있습니다.

이것은 산문에도 적용됩니다. Claude Code는 종종 마지막 문단에서 이전 문단을 요약하거나, 마지막 문장에서 이전 문장을 요약하려 합니다. 꽤 반복적일 수 있어서 삭제하거나 단순화하도록 요청해야 합니다.

## 팁 41: 자동화의 자동화

결국 핵심은 자동화의 자동화입니다. 더 생산적이 될 뿐만 아니라 과정을 더 재미있게 만드는 최고의 방법입니다.

같은 작업이나 명령어를 반복적으로 수행하는 자신을 발견할 때마다 - 몇 번은 괜찮지만 계속 반복한다면 - 전체 프로세스를 자동화하는 방법을 생각해보세요.

## 팁 42: 지식 공유하고 기여하기

가능한 한 많이 배우면, 주변 사람들과 지식을 공유할 수 있습니다. 게시물, 책, 강의, 비디오 등을 통해서요.

## 팁 44: dx 플러그인 설치하기

이 저장소는 `dx`(개발자 경험)라는 Claude Code 플러그인이기도 합니다. 위 팁들의 여러 도구를 한 번에 설치할 수 있습니다:

| 스킬 | 설명 |
|------|------|
| `/dx:gha <url>` | GitHub Actions 실패 분석 (팁 29) |
| `/dx:handoff` | 컨텍스트 연속성을 위한 인수인계 문서 생성 (팁 8) |
| `/dx:clone` | 대화 복제해서 분기 (팁 23) |
| `/dx:half-clone` | 컨텍스트 줄이기 위한 반복제 (팁 23) |
| `/dx:reddit-fetch` | Gemini CLI를 통한 Reddit 콘텐츠 가져오기 (팁 11) |
| `/dx:review-claudemd` | 대화를 검토해 CLAUDE.md 파일 개선 제안 (팁 30) |

**두 명령어로 설치:**

```bash
claude plugin marketplace add ykdojo/claude-code-tips
claude plugin install dx@ykdojo
```

**추천 동반 도구:** [Playwright MCP](https://github.com/microsoft/playwright-mcp) - `claude mcp add -s user playwright npx @playwright/mcp@latest`로 추가

## 팁 45: 빠른 설정 스크립트

이 저장소의 여러 권장 사항을 한 번에 설정하고 싶다면 설정 스크립트가 있습니다:

```bash
bash <(curl -s https://raw.githubusercontent.com/ykdojo/claude-code-tips/main/scripts/setup.sh)
```

스크립트는 구성할 모든 항목을 보여주고 건너뛸 수 있게 해줍니다:

```
설치:
  1. DX 플러그인 - 슬래시 명령어(/dx:gha, /dx:clone, /dx:handoff)와 스킬(reddit-fetch)
  2. cc-safe - 'rm -rf'나 'sudo' 같은 위험한 승인 명령어를 설정에서 스캔

설정 (~/.claude/settings.json):
  3. 상태 표시줄 - 화면 하단에 모델, git 브랜치, 커밋 안 된 파일, 토큰 사용량 표시
  4. 자동 업데이트 비활성화 - Claude Code 자동 업데이트 방지 (시스템 프롬프트 패치에 유용)
  5. MCP 도구 지연 로딩 - 필요할 때만 MCP 도구 정의 로드, 컨텍스트 절약
  6. Read(~/.claude) 권한 - clone/half-clone 명령어가 대화 기록을 읽을 수 있도록 허용
  7. Read(//tmp/**) 권한 - 프롬프트 없이 임시 파일 읽기 허용
  8. 출처 표시 비활성화 - 커밋에서 Co-Authored-By와 PR에서 출처 표시 제거

셸 설정 (~/.zshrc 또는 ~/.bashrc):
  9. 별칭: c=claude, ch=claude --chrome, cs=claude --dangerously-skip-permissions
 10. 포크 단축키: --fs를 --fork-session으로 확장 (예: claude -c --fs)

건너뛸 항목? [예: 1 4 7 또는 전체 설치는 Enter]:
```
